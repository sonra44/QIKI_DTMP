# RE_QIKI_Documentation_Completion_Criteria

## 1. Назначение

Этот документ фиксирует **не содержание проекта**, а правило, по которому можно честно сказать, что текущий документационный цикл по QIKI завершён.

Его задача — не допустить ложной завершённости в ситуации, когда:

- документов уже много;
- структура пакета уже собрана;
- архитектурная картина в целом восстановлена;
- но часть сильных утверждений ещё не доведена до достаточного уровня доказательности.

Иначе говоря, этот документ отвечает на вопрос:

**при каких условиях пакет можно перевести из состояния `assembled` в состояние `closed`.**

---

## 2. Главный принцип

Документация считается завершённой **не тогда, когда написано достаточно текста**, а тогда, когда одновременно выполнены четыре условия:

1. пакет собран как управляемая система;
2. ключевые сущности имеют явный статус;
3. сильные формулировки опираются на доказательную базу;
4. незакрытые зоны не спрятаны под видом канона.

Главное правило:

**closure нельзя объявлять сильнее, чем evidence.**

---

## 3. Два разных состояния: assembled и closed

## 3.1. Assembled

Состояние `assembled` означает, что:

- пакет уже существует как система;
- у него есть единый индекс;
- есть документы позиции, пути, зрелости, архитектуры, канона, продукта, evidence и risk-слоя;
- между документами можно восстановить текущую рабочую картину проекта.

Но при этом:

- runtime proof может оставаться частичным;
- active slice может оставаться в `proof-stage`;
- unresolved-зоны могут быть ещё не сняты;
- closure ещё не достигнут.

## 3.2. Closed

Состояние `closed` означает не просто наличие документов, а то, что:

- пакет собран;
- статусы синхронизированы;
- доказательный слой достаточен для текущего среза;
- незакрытые зоны явно названы и либо сняты, либо честно признаны как оставшиеся вне scope текущего closure.

## 5. Уровни честного завершения

## Level 1. Structural completeness

Пакет собран как система, а не как набор разрозненных текстов.

Критерий прохождения:

- существует полный индекс;
- существует текущая позиция;
- существует карта пути;
- существует матрица зрелости;
- существуют architecture / canon / product документы;
- существуют отдельные evidence и risk-документы.

## Level 2. Status completeness

У каждой ключевой сущности и у каждого ключевого документа есть явный статус.

Обязательные зоны:

- spec stack;
- runtime core;
- transport / contracts;
- decision layer;
- operator layer;
- audit layer;
- active slice;
- unresolved runtime proof items;
- duplicate-or-transition zones;
- suspected legacy zones;
- ownership ambiguity zones.

Критерий прохождения:

для каждой важной сущности известны:

- роль;
- статус;
- основной риск;
- следующий шаг.

## Level 3. Evidence completeness

Ключевые выводы опираются не только на аналитическую интерпретацию, но и на реальные основания.

Минимальный evidence-set:

- кодовой и конфигурационный слой;
- transport / contracts;
- runtime scenario logic;
- отдельные evidence notes по живым или частично живым проверкам;
- явная фиксация того, что **не** доказано.

Критерий прохождения:

каждое сильное утверждение можно привязать либо к коду, либо к runtime-evidence, либо честно ослабить.

## Level 4. Active-slice honesty

Текущий активный срез отражён честно и без симуляции закрытия.

Это означает:

- active slice назван явно;
- его статус не преувеличен;
- blocker не спрятан;
- следующий обязательный шаг определён.

Критерий прохождения:

по активному slice можно ответить на четыре вопроса:

1. Что реализовано?
2. Что не доказано?
3. Что мешает закрытию?
4. Что нужно сделать следующим действием?

## Level 5. Canon consistency

Канон проекта не противоречит сам себе.

Должны быть согласованы:

- архитектурный канон;
- продуктовая рамка;
- maturity/status система;
- ownership rules;
- event / transport semantics;
- runtime evidence;
- risk-layer.

Критерий прохождения:

в пакете нет скрытого конфликта, где:

- один документ объявляет ownership закрытым, а другой держит его unresolved;
- один документ объявляет active slice закрытым, а evidence держит его в proof-stage;
- один документ выдаёт продуктовую рамку за полностью доказанный runtime-факт.

## Level 6. Honest unresolved zones

Неопределённые и переходные области не замаскированы под канон.

Обязательные категории:

- `proof-stage`
- `unresolved`
- `duplicate-or-transition`
- `suspected-legacy`

Критерий прохождения:

каждая проблемная зона названа прямо и не спрятана за слишком сильной формулировкой.

## Level 7. Closure eligibility

Пакет может претендовать на closure только если предыдущие уровни выполнены одновременно.

Но даже этого недостаточно, если:

- evidence-layer явно сообщает о незакрытом blocker;
- maturity matrix держит active slice в `proof-stage`;
- core-docs содержат ещё не синхронизированные сильные утверждения.

---

## 6. Финальные условия документационного closure

Документационный этап по текущему срезу можно считать завершённым только если одновременно выполнены все условия ниже.

### Условие 1. Full package recovery
Есть единый индексный документ, через который можно восстановить весь рабочий пакет.

### Условие 2. Current-state recovery
Есть документ текущей позиции и документ полного аналитического пути, и они не противоречат друг другу.

### Условие 3. Status system recovery
Есть матрица зрелости, в которой ключевые сущности и документы имеют явные статусы.

### Условие 4. Core coherence
Есть architecture note, canon map и product/gate документ, не противоречащие maturity matrix и risk-layer.

### Условие 5. Evidence honesty
Есть отдельный evidence-документ, в котором текущий active slice либо доказан, либо честно обозначен как незакрытый.

### Условие 6. Risk honesty
Есть отдельный risk-документ, где blocker, ambiguity, duplicate и unresolved-зоны названы прямо.

### Условие 7. No hidden overclaim
В пакете нет сильных утверждений, которые evidence и maturity не могут поддержать.

### Условие 8. Closure traceability
По пакету можно ответить без прыжков между внешними заметками на пять вопросов:

1. Что это за проект?
2. Как он устроен?
3. Что реально работает?
4. Что доказано?
5. Что ещё не закрыто?

---

## 7. Что сейчас ещё не даёт считать цикл завершённым

На текущем этапе closure всё ещё блокируется как минимум следующими факторами:

- active slice всё ещё удерживается в `proof-stage`;
- `signature_changed live path` остаётся незакрытым доказательным элементом;
- runtime / compose operational state ещё требует достаточной повторной фиксации;
- часть ownership-границ остаётся unresolved;
- пакет уже синхронизирован намного лучше, чем раньше, но финальная closure-сверка ещё не исчерпана.

---

## 8. Что должно стать следующим обязательным шагом

Следующий обязательный шаг после этого документа — **не новый общий обзор**, а завершение closure-проверки через три действия:

1. повторная проверка runtime / compose state;
2. повторная проверка или честная переоценка `signature_changed live path`;
3. финальная синхронизация core-docs с maturity / evidence / risk-слоем.

Только после этого можно повторно проверить, достигнут ли documentation closure по текущему этапу.

---

## 9. Итоговая формула

Корректная формула текущего состояния такая:

```text
package structure = assembled
closure eligibility = partial
final closure = not yet
```

Именно эта формула должна использоваться до тех пор, пока evidence-layer и maturity matrix не позволят поднять статус без завышения уверенности.
