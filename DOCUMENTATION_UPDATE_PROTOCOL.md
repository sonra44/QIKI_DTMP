## 📋 ПРОТОКОЛ ОБНОВЛЕНИЯ ДОКУМЕНТАЦИИ

### ПРАВИЛО #1: Обязательные обновления при изменениях кода

**При ЛЮБОМ техническом изменении ОБЯЗАТЕЛЬНО обновить:**

1. **CLAUDE_MEMORY.md** - секция готовности компонентов
2. **CURRENT_STATE.md (snapshot)** - конкретные статусы
3. **Task‑досье в `TASKS/`** - реальные результаты

### ПРАВИЛО #2: Последовательность действий

**Каждая техническая задача ДОЛЖНА завершаться:**

```markdown
1. ТЕХНИЧЕСКОЕ ИЗМЕНЕНИЕ (код/конфигурация/исправления)
2. ВЕРИФИКАЦИЯ (тестирование изменения)  
3. ОБНОВЛЕНИЕ ДОКУМЕНТАЦИИ (синхронизация с реальностью)
4. CROSS-CHECK (проверка согласованности документов)
```

### ПРАВИЛО #3: Формат обновлений

**CLAUDE_MEMORY.md обновления:**
```markdown
**Последнее обновление:** [ДАТА] - [КРАТКОЕ ОПИСАНИЕ]
**Готовность [КОМПОНЕНТ]:** [ПРОЦЕНТ]% (было [СТАРЫЙ_ПРОЦЕНТ]%) - [ПРИЧИНА ИЗМЕНЕНИЯ]
```

**CURRENT_STATE.md (snapshot) обновления:**
```markdown  
### [КОМПОНЕНТ] ([ГОТОВНОСТЬ]%) [СТРЕЛКА] **[СТАТУС ИЗМЕНЕНИЯ]**
- **Проблема:** [ОПИСАНИЕ]
- **Решено:** [ДД.ММ.ГГГГ] - [ЧТО ИСПРАВЛЕНО]
- **Статус:** [АКТУАЛЬНОЕ СОСТОЯНИЕ]
```

## 💡 ПРИНЦИПЫ ЧЕСТНОЙ ДОКУМЕНТАЦИИ

### 1. РЕАЛЬНОСТЬ ПРЕВЫШЕ ПЛАНОВ
Документировать ЧТО ЕСТЬ, а не что планируется.

### 2. ПРОТИВОРЕЧИЯ = КРИТИЧЕСКИЙ БАГ  
Любое противоречие между документами требует немедленного исправления.

### 3. УСТАРЕВШИЕ ДАННЫЕ = ТЕХНИЧЕСКИЙ ДОЛГ
Документы старше 7 дней без обновлений требуют ревизии.

### 4. ПРОВЕРЯЕМЫЕ УТВЕРЖДЕНИЯ
Каждое заявление о готовности должно быть подтверждено практическим тестом.
