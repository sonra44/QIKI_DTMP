  - миссию можно пройти разными путями;
  - каждое ключевое решение игрока имеет измеримые последствия в телеметрии/состоянии мира.

  Почему это хороший старт:

  - минимальный overhead управления;
  - быстро видно, “игра уже живая или еще нет”;
  - просто держать фокус.

  ———

  2. 4 гейта
  Это более детальная версия, где риски разделяются раньше.

  1. G1: Физика + сенсоры

  - сначала доводится чистая “правда мира” и качество данных.

  2. G2: QIKI-арбитраж

  - отдельно закрепляется поведенческая логика QIKI и протоколы приоритета.

  3. G3: Бой

  - добавляется боевой контур как отдельный инженерный пакет.

  4. G4: Миссия + нарратив

  - собирается полноценная миссия и narrative-обвязка.

  Плюсы:

  - меньше смешения рисков;
  - легче диагностировать, где именно проблема.
    Минусы:
  - больше управленческой нагрузки;
  - выше шанс затянуть до “вечного прототипа”.

  Когда выбирать:

  - если команда больше 2-3 активных исполнителей;
  - если много параллельной разработки.

  ———

  3. Непрерывный backlog
  Без фиксированных гейтов, только приоритетный список задач.

  Как выглядит:

  - берется top-1 задача, делается, потом следующая;
  - периодически ревизия приоритетов;
  - нет жесткой “вехи” типа G1/G2/G3.

  Плюсы:

  - максимальная гибкость;
  - удобно при сильной неопределенности.
    Минусы:
  - легко потерять “форму продукта”;
  - сложно понять момент готовности;
  - высокий риск бесконечной полировки отдельных частей.

  Когда уместно:

  - очень ранний R&D;
  - когда вы еще не уверены в базовом цикле игры.

  ———

  Что это значит для вас practically

  1. взять 3 гейта как базовую рамку;
  2. внутри каждого гейта использовать mini-checklist из 5-10 проверок;
  3. после прохождения G1 решить, нужно ли дробить дальше до модели 4 гейтов.

  # Глобальный План: QIKI как Hard-Sci Fi Console Sim-Game (3 Gate Model)

  ## Summary

  Проект фиксируется как космический 3D симулятор с реальной физикой, где игрок видит только телеметрию/радар/сенсоры и взаимодействует с миром через QIKI.
  Фокус этапа: сначала рабочий техпрототип ядра, затем боевой контур, затем цельная миссионная вариативность при полной свободе действий и последствиях.

  ## Product Truth (канон)

  1. Это игра, не витрина платформы.
  2. Мир real-time, но slowdown/pause — часть механики.
  3. QIKI автономна, но ограничена протоколами/физикой/контекстом.
  4. Полная свобода игрока: можно менять цели, игнорировать задания, исследовать “что если”.
  5. Последствия решений обязательны и наблюдаемы в состоянии мира.
  6. Логика отказов QIKI адаптивна: объясняет причины, при повторной девиации сокращает ответы.
  7. Сенсорная правда:
      - WORKING_HEALTHY > точные данные
      - FAILED > данных нет + причина
      - DEGRADED (подавление/среда/перегрев/питание) > данные с пометкой недостоверности
      - OFF > намеренно отключен
      - N/A > только ошибка реализации (запрещено как штатный игровой статус)

  ## Gate Plan (decision-complete)

  ## G1 — Живой прототип ядра

  Цель: контур Observe > Ask/Command QIKI > Consequence без “фейковой” визуализации.

  In scope:

  1. Физический контур мира + объекты + события времени.
  2. Радар/сенсоры с полной моделью состояний и достоверности.
  3. Операторский UI (ORION) как единственная рабочая “кабина”.
  4. Командный контур QIKI: выполнимо/невыполнимо с контекстным объяснением.
  5. Slowdown/pause/time-scale (отдельно для simulation tick и QIKI reasoning budget).

  Out of scope:

  1. Многоцелевой бой.
  2. Полный нарративный пайплайн.

  DoD G1:

  1. Любая невозможная команда получает детерминированную причину (физика/зона/протокол).
  2. Все сенсорные состояния отображаются честно и различимо.
  3. Нет синтетических “красивых” значений без источника.
  4. Есть 30+ минут устойчивого runtime-smoke без рассинхрона состояния.

  ## G2 — Combat 1v1 + Протокольный арбитраж

  Цель: реализовать обязательный контур угрозы и базового боя.

  In scope:

  1. Encounter 1v1: обнаружение, выбор решения, атака/уклонение, последствия.
  2. Ограничения правил QIKI во время боя (запреты, зоны влияния, протоколы станции/сети).
  3. Энергия/перегрев/повреждения как реальные боевые ограничения.
  4. Изменение поведения QIKI при повторных риск-командах в одном контексте.

  DoD G2:

  1. Боевые решения меняют тактическое состояние и дальнейшие опции.
  2. Конфликт “команда игрока vs протокол” всегда объясняется и логируется причиной.
  3. После боя мир продолжает жить, а не “сбрасывается сценарием”.
  4. Все outcomes воспроизводимы при одинаковом seed/начальном состоянии.

  ## G3 — Миссионная вариативность и путь через понимание

  Цель: не линейная миссия, а вариативная система целей, где часть путей открывается через понимание условий мира.

  In scope:

  1. Миссии с несколькими путями и динамическими точками интереса.
  2. Возможность отклониться от задания и породить скрытые ветви событий.
  3. Нелинейная достижимость: часть целей недостижима до открытия условий.
  4. “Fail-forward” логика: провал в одной ветке порождает другие возможности, не только game-over.

  DoD G3:

  1. Минимум 1 большая миссионная дуга с вариативными маршрутами решения.
  2. Минимум 3 устойчивых типа исходов: успешный, компромиссный, тяжелый.
  3. Игрок может осознанно менять курс и получать системно объяснимые последствия.
  4. Поведение QIKI и мира не “скриптуется вручную” под единственный канонный путь.

  ## Public Interfaces / Contracts (обязательные изменения)

  ## Command/Response

  1. qiki.intents.v1 — команда/запрос игрока к QIKI
     Поля: intent_id, operator_text, context_snapshot_ref, priority_hint, timestamp.
  2. qiki.responses.v1 — ответ QIKI
     Поля: intent_id, status (accepted|rejected|deferred|blocked), reason_code, human_explanation, suggested_conditions[], planned_actions[].

  ## World/Combat Telemetry

  1. qiki.world_state.v1 — базовое состояние мира/корабля/ресурсов.
  2. qiki.sensor_state.v1 — состояние сенсоров + доверительный диапазон.
  3. qiki.combat_event.v1 — события encounter 1v1.
  4. qiki.protocol_block.v1 — формализованная запись блокировок протоколами.

  ## UI Contract (ORION)

  1. Единая зона “Action Legality”: почему можно/нельзя выполнить действие.
  2. Единая зона “Sensor Trust”: цвет + числовой confidence corridor.
  3. Явные статусы времени: RUNNING / SLOWED(xN) / PAUSED.

  ## Test Cases and Scenarios

  ## G1 tests

  1. Команда связи вне зоны: deterministic reject + корректная причина.
  2. FAILED сенсор: данных нет, причина видна, ложные значения отсутствуют.
  3. DEGRADED сенсор: данные есть + error bounds + trust color.
  4. Time-scale: одинаковый старт > одинаковый исход по инвариантам.

  ## G2 tests

  1. Encounter 1v1 end-to-end: detect > decide > execute > consequence.
  2. Протокольный запрет в бою: блок + объяснение + альтернативы.
  3. Пределы ресурса: энергия/перегрев/повреждение влияют на разрешенные действия.
  4. Повтор девиантной команды: укорачивание ответа QIKI без потери причинности.

  ## G3 tests

  1. Одна цель — минимум 2 валидных маршрута достижения.
  2. Отклонение от “официального” задания рождает скрытое событие.
  3. Невозможная цель становится возможной после выполнения условий.
  4. Провал ветки не ломает сессию, а меняет контекст мира.

  ## Multi-Agent Parallel Plan (как делаем быстрее)

  1. Agent A (Simulation Core): physics/timing/world-state contracts.
  2. Agent B (QIKI Logic): intent arbitration, protocol blocks, response policy.
  3. Agent C (ORION UX): legality panel, trust panel, time-state rendering.
  4. Agent D (Verification): deterministic tests, scenario packs, evidence reports.

  Правило синхронизации:

  1. Ежедневный contract-freeze по схемам сообщений.
  2. Никаких merge без прохождения cross-agent integration smoke.
  3. Любая новая механика сначала через contract PR, потом implementation PR.

  ## Assumptions and Defaults

  1. Default режим данных: честные реальные значения + явная недостоверность (без игровых “подрисовок”).
  2. Default поведение QIKI: автономная, но причинно-объяснимая и протокольно ограниченная.
  3. Default прогресс: техпрототип (G1) > расширение (G2/G3), без обещаний “полного релиза”.
  4. Default риск-политика: лучше отказ с объяснением, чем молчаливое выполнение опасной/невозможной команды.

  # Глобальный План v2: ORION/QIKI с Инженерным Мониторингом (по вашему hardware-листу)

  ## Summary

  План фиксирует игру как хардкорный космо-sim: real-time мир, пауза/замедление как механика, QIKI с автономией и протокольными ограничениями, полная свобода действий игрока с реальными последствиями.
  Ваш список телеметрии принят как целевой операторский канон и разложен по 3 гейтам реализации.

  ## UX Canon (обновлено)

  1. Интерфейс обязан поддерживать классический remote craft monitoring:
      - питание, заряд/разряд, источники/потребители, шина, лимиты, fault-состояния;
      - орбитальная и навигационная кинематика;
      - ADCS/AACS, thermal, propulsion, comms, compute/safety.
  2. Экран Power/EPS по умолчанию: Glance + Expand + Explain.
  3. Политика единиц: SI в источнике + автоконверсия в UI с явной подписью единиц.
  4. Вычисляемые орбитальные параметры (apo/peri/period/eccentricity) разрешены, но только с confidence/quality метками.

  ## Truth/Quality Contract (обязательный)

  Каждое поле несет:

  - value
  - state: healthy | degraded | failed | off
  - quality: green | yellow | orange | red
  - confidence_pct (0..100)
  - reason_code (если не healthy)
  - source (sensor|estimator|derived)
  - updated_at

  Правила:

  1. healthy = точные данные.
  2. failed = значения нет, показываем причину.
  3. degraded = значение есть, но с коридором ошибки и цветом доверия.
  4. off = подсистема отключена намеренно.
  5. N/A = только ошибка реализации/контракта (не нормальный игровой статус).

  ## Gate Plan

  ## G1 — Живой инженерный прототип

  Цель: полный контур мониторинга + QIKI-команды + последствия без декоративных данных.

  Включить:

  1. Полные панели Navigation, Power/EPS, Thermal, Comms basic, Safety/Compute.
  2. Сенсорный state/quality contract для всех ключевых полей.
  3. Командный контур QIKI с контекстным объяснением невыполнимости.

  DoD:

  1. Любая “невозможная” команда имеет объяснимую причину.
  2. Нет скрытых аппроксимаций без source/confidence.
  3. 30+ минут стабильного runtime-smoke.

  ## G2 — Combat 1v1 + Protocol Arbitration

  Цель: боевой контур при сохранении инженерной достоверности.

  Включить:

  1. Threat detect -> решение -> действие -> последствия.
  2. Протокольные блоки QIKI в бою (с reason_code).
  3. Влияние боя на EPS/Thermal/Propulsion/Comms.

  DoD:

  1. Исход боя отражается в телеметрии и ограничениях.
  2. Конфликты “команда vs протокол” детерминированы и объяснимы.

  ## G3 — Вариативная миссионность при полной свободе

  Цель: нелинейный сценарный слой поверх системной симуляции.

  Включить:

  1. Динамические точки интереса.
  2. Скрытые ветви при отклонении от “основного” задания.
  3. Fail-forward outcomes вместо чисто линейного скрипта.

  DoD:

  1. Миссия поддерживает разные пути достижения/недостижения целей.
  2. Игрок может менять курс и получать системно логичные последствия.

  ## Public Interfaces / Schema Changes

  Добавить/закрепить:

  1. qiki.telemetry.v2 (или schema_version=2 в текущем потоке):
      - унифицированные поля + state/quality/confidence/source/reason_code.
  2. qiki.derived.orbit.v1:
      - apo/peri/ecc/period/inc + метод расчета + confidence.
  3. qiki.protocol_block.v1:
      - формальные записи запретов/ограничений команд.
  4. qiki.combat_event.v1:
      - боевые события и их причинная связь с системами.

  ## Test Cases

  1. Sensor state matrix: healthy/degraded/failed/off для каждого критичного поля.
  2. Degraded data: UI показывает value + confidence + quality color + reason.
  3. Protocol block: команда блокируется корректным reason_code.
  4. Orbit derived: вычисляемые метрики имеют source=derived и confidence.
  5. EPS stress: перегрузка/дефицит мощности корректно отражается в Explain.
  6. Combat 1v1: от обнаружения до последствия с трассируемой телеметрией.

  ## Assumptions / Defaults

  1. Значения по умолчанию не “дорисовываются”.
  2. UI всегда помечает происхождение и надежность данных.
  3. QIKI может отказать жестко; краткость/подробность ответа зависит от контекста и повторов.
  4. Полная свобода игрока сохраняется, но мир отвечает физикой, протоколами и риском.
