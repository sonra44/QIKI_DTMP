# ADR: Владелец тика и решений = qiki-dev (мозг №1); intents — read-only проекция

**Status:** ACCEPTED (design-only; K4-консолидация после K2)
**Date:** 2026-07-10
**Decision owner:** оператор + CLI-агент (карта `docs/dev/AUDIT_2026-07-10_STRUCTURE.md`, S2/N2/N6)

## Context

- Код-правда: два мозговых процесса гоняют один конвейер независимо —
  `qiki-dev` (`run_tick`) и `q-core-intents`
  (`_refresh_agent_snapshot`): отдельные QCoreAgent + WorldModel, оба
  тянут q-sim по gRPC, оба строят треки и оценивают гварды (S2).
- Compose уже признаёт иерархию, но она не закреплена решением:
  «Волна 0 Ф1: intents не владеет FSM-правдой; …Fallback ЯВНО помечен»
  (`docker-compose.phase1.yml`, env intents; проверено live).
- Стабильный факт архитектуры: канонический FSM-источник phase1 —
  `QIKI_FSM_SOURCE=local_mainfsm` (CLAUDE.md, Stable Architecture Facts).
- Отягчающие: в мосте спит ПОЛНЫЙ второй intent-исполнитель
  (`_proposal_store` + `handle_qiki_intent` + ACCEPT→COMMANDS_CONTROL),
  выключенный одной env-строкой, подписчики без queue-group — опечатка
  в env = двойное исполнение intents (N2); гвард-тревога считается
  дважды — мозгом (влияет на FSM) и мостом (event на экран) по разным
  идентичностям (N6); refresh intents блокирует event loop (дефект 0.8
  Блока 0).
- Канон-инвариант: dual-source решений запрещён без явного ADR
  (`CONTEXT_LOCK_QIKI_DTMP.md` п.5). Этот документ и есть такой ADR.

## Decision

1. **Владелец тика и решений = qiki-dev (мозг №1, `run_tick`).**
   FSM-правда, гварды и треки-для-решений принадлежат только ему.
2. **intents = проекция, владелец ДИАЛОГА.** Обслуживает LLM/F5-контур
   и intents; его world-view — read-only проекция, непригодная для
   FSM/гвардов. Целевое состояние: intents потребляет снапшоты мозга
   №1; до K4 собственный refresh остаётся, но его выводы не могут
   становиться решениями.
3. **Мостовой intent-исполнитель (N2) — нештатный.** Двойное
   исполнение intents запрещено как инвариант; путь в K4: снести
   второй контур либо заменить env-флаг жёстким guard'ом + queue-group
   (env-строка не является механизмом безопасности).
4. **Гвард-тревога считается один раз — мозгом №1.** Мостовая
   `radar_guard_cadence` — отображение, не решение; после ADR
   «идентичность целей» обе стороны говорят об одних track_id.

## Consequences

Positive:
- Один орган решений — согласовано с Safe Mode authority (Q-Core) и
  запретом dual-source; S2 получает канонное закрытие, K4 становится
  механической работой, а не спором.
- N2 из «на волоске» переходит в явный долг с зафиксированным
  направлением.

Negative / costs:
- До K4 сохраняется двойной вычислительный налог (два gRPC-потока,
  двойная сборка треков).
- intents-диалог живёт на проекции — задержка снапшота между мозгами
  должна быть честно видима (age), не маскироваться.

## Alternatives considered

1) **intents-владелец тика** — отвергнут: compose сам признаёт
   обратное; блокирующий sync-refresh (0.8) делает его непригодным
   владельцем реального времени.
2) **Третий выделенный процесс-владелец** — отвергнут: умножает мозги
   (анти-S2), phase1 не требует.
3) **Только queue-group на подписчиках** — лечит симптом N2 (дубль
   доставки), но не закрепляет владение решением; недостаточно.

## Пересечения (сверка «задним числом»)

- Тик ≠ диалог: F5-контур остаётся за intents — этот ADR не трогает
  владение беседой/persona (DECISIONS id=5111).
- N3 (три сущности «fsm_state») — смежная гигиена имён: sim run-state
  (STOPPED/RUNNING) не есть FSM мозга; переименование — отдельный
  шаг за пакетом, здесь только фиксация различия.
- Пауза/slowdown мира — собственность sim run-state, не тика мозга;
  pause-честность (этап после 10) не конфликтует.
- Порядок по карте: K2 (идентичность) → K4 (консолидация мозгов,
  снимает N2/N6 физически). Этот ADR даёт K4 критерий приёмки.
