## Назначение

Этот файл является единым каноническим источником истины для следующего продуктово-критичного этапа после честного закрытия `G1-QIKI-001`.

Если предыдущий этап доказал базовый цикл:
- `Наблюдение -> Запрос/Команда QIKI -> Legality/Trust -> Consequence`,

то новый этап обязан доказать следующий уровень игры:
- `Наблюдение -> Запрос цели/намерения -> План QIKI -> Явное исполнение шагов -> Наблюдаемое движение во времени`.

Проект трактуется так:
- это hard sci-fi console sim-game,
- ORION V является операторской кабиной,
- QIKI является автономным, но протокольно и физически ограниченным агентом,
- время является частью механики, а не только фоном,
- оператор должен видеть не только ответ, но и исполнение сценария.

## Почему именно этот этап теперь главный

После `G1-QIKI-001` уже доказано:
- ORION V показывает legality/trust/consequence,
- QIKI умеет блокировать, откладывать и подтверждать действия,
- есть путь `pending -> confirm -> confirmed`,
- `F1 quick-actions` и их логика проверены unit/headless/live proof.

На текущем этапе уже дополнительно доказано:
- первый procedural scenario проходит через существующий `ProcedureEngine`,
- ORION V показывает `Plan Preview` и `Execution State`,
- `F1` теперь содержит отдельную операторскую секцию `Процедура/Procedure`,
- из `F1` есть прямой переход `Процедуры/Procedures -> F6` в журнал procedural audit.
- live runtime-proof подтвердил этот слой на поднятом Docker stack.
- второй procedural scenario доказал reduced-speed path (`x0.25`) без нового transport.

Но всё ещё недореализован следующий критичный контур из `LOG.MD`:
- QIKI не просто отвечает на локальный запрос,
- QIKI должна уметь брать намерение оператора и разворачивать его в исполняемый сценарий,
- оператор должен видеть, как меняется мир во времени,
- пауза/замедление должны стать частью принятия решения, а не просто служебной функцией.

Без этого проект остаётся на уровне “умный операторский терминал”, но не выходит на уровень “живой QIKI-centric sim-game”.

## Имя этапа

`G1-QIKI-002: Procedural Execution + Time Control`

## Цель

Сделать первый законченный контур, где QIKI не только оценивает допустимость, но и готовит/запускает управляемую процедуру, а ORION V показывает:
- план,
- текущее исполнение,
- контроль времени,
- подтверждённые переходы между шагами.

## Продуктовая истина этапа

1. QIKI может быть автономной, но не магической.
2. Намерение оператора должно раскладываться в понятные шаги.
3. Если действие длительное, оператор должен видеть не только итог, но и ход исполнения.
4. Время игры должно быть частью механики:
   - real-time по умолчанию,
   - pause/slowdown как осмысленный инструмент,
   - у QIKI и оператора могут быть разные “масштабы времени”, но это должно выражаться через реальные состояния и протоколы.
5. ORION обязан показывать, что происходит сейчас:
   - готовится,
   - выполняется,
   - ждёт подтверждения,
   - завершено,
   - прервано,
   - заблокировано.

## В scope

1. Один канонический QIKI-сценарий с несколькими исполняемыми шагами.
2. Явная поверхность плана/процедуры в ORION V.
3. Видимый execution-state по шагам.
4. Подтверждённая интеграция с существующим `ProcedureEngine` и `sim.*` control path.
5. Использование pause/start/speed как части доказуемого операторского цикла.
6. Docker-first proof и отдельный runtime-proof.

## Вне scope

1. Новый combat loop.
2. Новый mission system.
3. Новый radar renderer.
4. Новый большой lore layer.
5. Полировка UI вне нового procedural path.

## Обязательные UX-поверхности

### 1. Plan Preview

До запуска процедуры ORION обязан показать:
- что именно QIKI собирается делать,
- из каких шагов состоит сценарий,
- какие шаги уже авторизованы,
- какие шаги требуют отдельного подтверждения.

### 2. Execution State

Во время процедуры ORION обязан показывать:
- имя процедуры,
- номер текущего шага,
- общее число шагов,
- состояние (`pending/running/waiting_confirm/confirmed/failed/aborted/done`),
- последнюю причину задержки или отказа.
- В MVP это должно быть видно не только внутри `QIKI`-блока, но и как отдельная системная секция `Процедура/Procedure` в `F1`.

### 3. Time Control Surface

ORION обязан давать оператору ясный доступ к:
- `sim.pause`,
- `sim.start`,
- скорости исполнения (`speed`), если она уже поддерживается runtime.

И обязан показывать:
- текущее sim-state,
- влияет ли pause на исполняемую процедуру,
- не продолжает ли мир двигаться скрыто.
- В MVP это должно читаться на одном экране `F1` вместе с procedural state, а не только через вспомогательные команды.

### 4. Consequence Over Time

После запуска процедуры должны быть видимы:
- ACK по шагам,
- изменение telemetry/sim_state,
- прогресс процедуры,
- честный итог: `done`, `failed`, `aborted`.

## Канонический первый сценарий этапа

Первым сценарием этапа выбирается не бой и не миссия, а безопасный процедурный контур:

`QIKI, подготовь безопасную стабилизацию наблюдения`.

В инженерном MVP-переводе это означает:
- QIKI готовит детерминированную процедуру `safe_pause_resume` или близкий по смыслу сценарий на уже существующем `ProcedureEngine`;
- ORION V показывает план шагов;
- процедура использует существующие `sim.pause` / `sim.start` ACK-пути;
- оператор видит шаги, паузу, подтверждённое выполнение и возврат к нормальному состоянию.

Причина выбора:
- этот путь уже имеет реальную опору в коде (`ProcedureEngine`, `sim.pause/start`, audit stream);
- не требует выдумывать новую физику;
- вводит в игру именно то, чего сейчас не хватает: наблюдаемое исполнение во времени.

## Definition of Done

Этап считается завершённым, когда:
1. один многошаговый QIKI-сценарий реализован как процедура;
