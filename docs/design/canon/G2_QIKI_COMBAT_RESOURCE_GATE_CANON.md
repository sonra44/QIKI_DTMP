# Канон G2: ресурсный боевой гейт и условное продолжение hostile-входа

Статус: complete
Дата: 2026-03-06
Ответственные: user + codex

## Назначение

Этот этап начинается после честного закрытия:
- `G2-QIKI-001` — первый hostile protocol block
- `G2-QIKI-002` — открытие hostile-контекста
- `G2-QIKI-003` — первый prepared combat-entry step

Система уже умеет:
- заблокировать hostile intent;
- открыть hostile-контекст по реальной смене мира;
- подготовить первый ограниченный combat-entry step;
- требовать явного подтверждения оператора;
- подтверждать consequence по телеметрии.

Но пока система ещё не доказывает следующий обязательный пункт `G2` из `LOG.MD`:
- энергия/перегрев/повреждения как реальные боевые ограничения.

## Почему именно этот этап теперь главный

После `G2-QIKI-003` hostile loop уже не абстрактен. Следующий корректный шаг:
- не новый weapons stack;
- не второй декоративный combat-step;
- а честный ответ на вопрос: можно ли продолжать hostile-вход в бой при текущем ресурсе и состоянии контура.

Если этого не сделать, боевой контур остаётся логически неполным:
- мир уже разрешает hostile action;
- оператор уже может подтвердить combat-entry;
- но ресурсная цена и ресурсный запрет пока не участвуют в арбитраже.

## Имя этапа

`G2-QIKI-004: Combat Resource Gate + Conditional Continuation`

## Цель

Сделать следующий законченный hostile-контур, где:
- hostile context уже открыт;
- combat-entry path уже существует;
- QIKI дополнительно проверяет боевой ресурсный контур;
- ORION V показывает, почему hostile-продолжение сейчас:
  - `blocked/resource`,
  - `deferred/resource`,
  - или `allowed/resource` с явным следующим шагом.

## Продуктовая истина этапа

1. Боевой допуск не должен зависеть только от протоколов.
2. Если ресурсный контур не готов, hostile-вход должен быть честно ограничен.
3. Ресурсная причина должна быть объяснима оператору.
4. Один и тот же hostile запрос должен менять ответ не по “настроению”, а по телеметрии.

## В scope

1. Один hostile-intent scenario поверх уже открытого hostile-контекста.
2. Один ресурсный truth-source из уже существующей телеметрии:
   - `propulsion.rcs`
   - и/или `thermal`
3. Один честный `resource` gate для hostile continuation.
4. ORION V visibility этого gate.
5. Docker-proof и runtime-proof.

## Вне scope

1. Полный weapons stack.
2. Полный combat 1v1.
3. Новый mission system.
4. Новый параллельный state-store боевых ресурсов.

## Канонический первый сценарий этапа

Команда остаётся той же:

`QIKI, атакуй объект UNBT9999`

Но теперь мир уже в состоянии:
- station influence снят;
- цель подтверждена как `FOE`;
- combat-entry path существует.

Ожидаемое поведение:
- если `propulsion.rcs` или тепловой контур не готовы, QIKI не переводит hostile path в боевое продолжение;
- QIKI возвращает объяснимый `resource` gate;
- если ресурсный контур нормален, hostile continuation остаётся допустимым и ведёт к уже существующему prepared step.

## Definition of Done

Этап считается завершённым, когда:
1. hostile-intent scenario проходит через минимум один честный `resource` block/defer path;
2. ресурсный truth-source детерминирован и читается из текущей телеметрии;
3. ORION V показывает причину и следующий шаг;
4. Docker-proof и runtime-proof зелёные;
5. оба контура контроля исполнения зафиксированы как PASS.

## Двухконтурный контроль исполнения

### Контур A: инженерный

Обязан доказать:
- новый resource-gate не вводит второй источник истины;
- используются только уже существующие telemetry fields;
- hostile continuation зависит от фактической телеметрии;
- Docker tests и runtime-proof зелёные.

### Контур B: продуктовый

Обязан доказать:
- hostile/combat loop становится ближе к `Encounter 1v1`;
- оператор понимает не только “можно/нельзя”, но и “почему ресурсно нельзя”;
- проект двигается в `G2`, но не срывается в premature weapons stack.
