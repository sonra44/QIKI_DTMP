# ADR-0019 — Бортовой каталог модулей: источник паспортов для параметризованной установки (v2)

## Status

**Accepted** (оператором 2026-07-05, после трёхлинзовой ревизии: прайор-арт
10 игр, adversarial-ревью кода/канона — 9 CONFIRMED-дефектов, геймдизайн-сверка
с G1-циклом; v1 переработан в v2 по её вердикту).

## Date

2026-07-05 (v1 утром, v2 после ревизии вечером).

## Context

B2 (ADR-0018) подключил мост M7-M9 к живому контуру, но policy жёстко предлагает
единственный модуль `test_sensor_module_001@F06`, контроллер умеет только
самопроверку, а runner вызывает её без аргументов — параметры пломбы до тела
не доезжают (сегодня невидимо: параметры совпадают с константами; при наивной
параметризации исполнялось бы НЕ одобренное). Для геймплейной параметризации
(«установи *антенну связи* на *F03*») нужны: источник правды «какие модули есть
на борту», контракт доведения параметров пломбы до тела и игровые свойства
выбора (дефицит, необратимость, различие граней).

RAG-gate / repo-проверка 2026-07-05 (repo — финальный источник):

- **Канон** `06_INTERFACE_CONTROL.md` §10 (IF-MODULE-PASSPORT-001,
  canon/template-only): модуль до runtime-ready проходит паспортную проверку;
  §10.6 — полный целевой состав полей; §10.8 — словарь reason-кодов;
  §6: блокировка всегда с reason_code, немой отказ недопустим.
- **Код** (`q_core_agent/core/body_structure.py`): `ModulePassport` —
  минимальная форма slice 0001 (`module_id, module_class, mount_point,
  provided_capabilities`); `run_attach_pipeline` принимает произвольный запрос
  с паспортом и уже несёт guard'ы `MOUNT_POINT_OCCUPIED` /
  `MOUNT_POINT_UNKNOWN` / class-forbidden; `FACE_IDS` = `F00..F11`;
  `face_mount_classes` — allowed/forbidden классы по граням (fixture,
  помечен «NOT canon, placeholder»).
- Каталога модулей в repo нет (`config/modules/` отсутствует).
- Mount `./config` подтверждён в контейнерах policy и консоли (один
  хост-каталог — версии файла не расходятся; расходиться может момент чтения).

## Decision

### 1. Каталог — один файл, один загрузчик, fail-closed

`config/modules/catalog.json` — единственный источник паспортов доступных
бортовых модулей (лор: грузовой отсек — «что есть, то и ставим»). Читают его
policy и консоль **одним общим модулем-загрузчиком** (shared-код), **per-request**
(файл мал; атомарность момента чтения важнее кэша — никто не владеет устаревшей
копией).

Fail-closed каталога: нечитаемый/пустой JSON → `CATALOG_UNAVAILABLE`;
кривая запись, дубликат `module_id`, неизвестный `module_class` →
`CATALOG_INVALID`. В обоих случаях attach-кандидаты — `deferred` с кодом
(канон §6: немой отказ недопустим).

### 2. Запись каталога и пломба на весь паспорт

Запись = паспорт-шаблон в КОДОВОЙ (минимальной) форме **без `mount_point`** +
операторские поля:

```
module_id, module_class, provided_capabilities,
display_name_ru, quantity
```

`mount_point` подставляется при установке из выбора оператора.
`display_name_ru` живёт ТОЛЬКО в UI-слое; evidence/audit несут `module_id` и
коды (DISPLAY_CANON: коды кодами). `display_name_en` не вводится (игровое поле
русское; en-имя ничего не даёт).

**Пломба покрывает весь паспорт**: policy кладёт полный шаблон записи в
`parameters` кандидата → M5 пломбирует `module_id + mount + module_class +
provided_capabilities + quantity-требование`. Правка catalog.json между approve
и исполнением (файл операторо-редактируем by design) не может подменить
исполняемое: тело собирает паспорт из ПЛОМБЫ, не перечитывает каталог.

### 3. Контракт исполнения — параметры только из пломбы

Контроллер получает новый API `attach_module(module_id, mount, passport)`;
`_body_attach_runner` строит вызов **только из запломбированного решения**.
RED-тест обязателен: «параметры пломбы == параметры, дошедшие до
`ModuleAttachRequest`».

Сопутствующие фиксы контроллера (без них P3 невозможен):
- **шорткат `already_attached` убирается**: занятость гнезда проходит через
  конвейер (guard `MOUNT_POINT_OCCUPIED` уже есть) — честный отказ с СВОИМ
  аудит-событием, никаких stale `audit_event_id`;
- **`can_run` не запирается** после успешной установки (допустимость решает
  конвейер, не снапшот) — иначе второй модуль не поставить;
- **`reset()` пишет аудит-событие** (сейчас — молчаливый массовый detach,
  нарушение §10.10).

### 4. Разделение правды и единый словарь кодов

- policy валидирует по каталогу и списку граней: модуль не в каталоге →
  `MODULE_UNKNOWN` (**policy-layer код**, осознанно вне канонного словаря
  §10.8); гнездо вне граней → **реюз body-кода `MOUNT_POINT_UNKNOWN`**
  (третье имя не вводится); `quantity == 0` → `MODULE_DEPLETED` (policy-layer);
- `FACE_IDS` policy **импортирует из `body_structure`** (оба в q_core_agent) —
  своей копии списка граней не заводит;
- классовая совместимость грани и занятость гнезда — runtime truth КОНВЕЙЕРА;
  policy их не дублирует и не предсказывает. Trade-off зафиксирован: кандидат
  «запрещённый класс на грань» дойдёт до отказа тела — это честно (правда у
  тела) и это игровая информация; нон-авторитетная подсказка policy — отложена
  (см. «Отложено»).

### 5. Игровые свойства (сошлись две линзы ревизии)

- **Дефицит**: `quantity` — считанные экземпляры; списание рантайм-леджером
  отсека по `effect=confirmed` (НЕ по ACK — ADR-0015);
- **Заявленная необратимость**: в предпросмотре решения строка
  «снятие невозможно»; detach — отдельный трек отдельным ADR;
- **Грани разные** (данные fixture, не код): `face_mount_classes` варьируется
  по граням (пример: antenna — только F00/F06) — появляется головоломка
  размещения и повод спрашивать бота;
- **Стартовый состав отсека** (5 записей):
  `test_sensor_module_001` (sensor, quantity 2),
  `comm_antenna_module_001` (antenna, 1),
  `science_probe_module_001` (science, 1),
  `rcs_cluster_module_001` (rcs-cluster, 1 — forbidden-класс: «есть в грузе,
  телу ставить нельзя»; делает forbidden-путь тестируемым и игровым),
  `salvage_sensor_damaged_001` (sensor, 1, битый паспорт — лор: манифест
  повреждён радиацией Терты) → `MODULE_PASSPORT_INVALID` при попытке;
- **Deferred-окно** по теплу/питанию (fail-closed ADR-0018) — обязательный
  live-сценарий P4: «попробуй, когда остынет» (G1: условие допустимости).

### 6. Гейты не меняются

Пломба M5 (теперь на весь паспорт), approve M6, fail-closed предусловия и мост
ADR-0018 — как есть. Полный канонный состав паспорта §10.6 НЕ заявляется
(template-only остаётся); расширение — отдельный трек. Персистентность тела
между рестартами консоли не заявляется (прототип, in-memory контроллер).

## Roadmap (после Accepted; каждый P = патч + тесты + live-proof)

- **P1**: shared-загрузчик каталога + fail-closed коды; policy отвечает
  списком доступных модулей (без выдумок, только каталог; quantity виден).
- **P2**: парсер «установи <модуль> на <гнездо>»; **матч-контракт: только по
  `module_id` или однозначному префиксу id/класса — НЕ по свободному
  `display_name_ru`** (поведение policy не может зависеть от
  операторо-редактируемой строки); валидация по §4; кандидат несёт полный
  шаблон в parameters (§2).
- **P3**: `attach_module(module_id, mount, passport)` в контроллере + фиксы §3;
  RED-тест пломба==тело ПЕРВЫМ; леджер quantity.
- **P4**: live-proof полного цикла + негативы: `MODULE_UNKNOWN`,
  `MOUNT_POINT_OCCUPIED` (через конвейер, не шорткат), forbidden-класс
  (rcs_cluster), `MODULE_PASSPORT_INVALID` (битый паспорт), `MODULE_DEPLETED`
  (quantity), deferred-окно тепло/питание. Каждый отказ — с разборчивой
  причиной оператору (антипаттерн EVE «не влезло без арифметики» запрещён).

## Rejected alternatives

- Пломба только `module_id/mount` (v1) — отклонено: TOCTOU на
  операторо-редактируемом каталоге между approve и исполнением.
- Наивный P3 «тот же self-check» — отклонено: runner игнорирует параметры
  пломбы; исполнялось бы не одобренное при зелёных гейтах.
- Файл-на-модуль (`config/modules/*.json`) — отклонено для прототипа:
  один catalog.json атомарнее; дробление — при внешнем пополнении.
- Каталог в policy-константах — отклонено: данные не код; составом отсека
  владеет оператор.
- Дублирование class-правил/FACE_IDS в policy — отклонено: два владельца =
  дрейф (правда у тела; FACE_IDS — импорт).
- Матч intent по `display_name_ru` — отклонено: поведение policy стало бы
  функцией редактируемой UI-строки.
- Кэш каталога от старта сервиса — отклонено: два читателя с разными моментами
  чтения = рассинхрон policy/консоль.

## Отложено осознанно (отдельные треки/ADR)

- **Установка как многошаговая процедура с осложнениями-развилками**
  (Duskers/Stationeers; сильнейшая рекомендация прайор-арта) — кандидат на
  следующий большой трек; требует своего ADR (на грани «без новой физики»).
- Query-intent «доложи гнёзда/отсек» (информационная асимметрия как игра).
- Нон-авторитетная подсказка policy о классах граней (hint, не gate).
- Износ/деградация установленного; отложенные атрибутируемые последствия.
- Payoff способностей (антенна реально улучшает связь и т.п.).
- Полный паспорт §10.6 (масса/power/thermal-профили); detach-цикл.
- Миграция словаря `module_class`, когда приедет канонный Face Map
  (текущий fixture честно помечен placeholder).
