# Структурный аудит 2026-07-10: дубли владельцев и параллельные пути

Два Serena-first разведчика (консольный + сервисный контуры), main=9486295.
Все находки подтверждены file:line. Префиксы: S- сервисный контур, C- консоль.

## ЯДРО (самое острое)

### S1. ТРИ независимых радар-матчинга; оператор и мозг видят РАЗНЫЕ цели
| Реализация | Пороги ассоциации | Живость в phase1 |
|---|---|---|
| мозг `q_core_agent/core/world_model.py:242` | 2500 м / 12° / 8° / 250 м/с | ЖИВОЙ ×2 (оба мозга) |
| мост `faststream_bridge/radar_track_store.py:73-186` | 12 м евклид / 15 м/с | ЖИВОЙ (RADAR_FRAMES→RADAR_TRACKS) |
| fusion `q_core_agent/core/radar_pipeline.py`+`radar_fusion.py` | gate + max_age 2.0 c | LEGACY (только mission_control/session_server, вне compose) |

Ключевое следствие: **консоль ORION V подписана на мостовые треки
(RADAR_TRACKS), а FSM-гварды считаются по мозговым трекам** — track_id
несравнимы, мост вдобавок схлопывает кадр в ОДИН трек
(radar_handlers.py:44-52), мозг держит все. Оператор видит одну
идентичность целей, бот решает по другой.

### S2. ДВА мозговых процесса гоняют один конвейер независимо
`qiki-dev` (run_tick) и `q-core-intents` (_refresh_agent_snapshot,
intents:4330-4336) — отдельные QCoreAgent+WorldModel, оба тянут q-sim по
gRPC, оба строят треки и оценивают гварды/пропозалы. compose:247 сам
признаёт «intents не владеет FSM-правдой», но код-владелец тика не
закреплён.

### S3. Три шкалы «смерти трека»
shared radar_freshness 5/30 c (мозг+консоль) vs fusion max_age 2.0 c vs
мост max_misses=3 (вообще не время). Докстринг radar_freshness заявляет
«единую шкалу» — мост и fusion её не используют.

## Карта движения данных (phase1)

```
q-sim (world_model.get_state — единый источник)
 ├─ gRPC → МОЗГ №1 (qiki-dev) и МОЗГ №2 (intents) — каждый свои треки/гварды (S1а, S2)
 ├─ NATS RADAR_FRAMES → мост RadarTrackStore (S1б) → RADAR_TRACKS → консоль
 ├─ NATS телеметрия (без field_sources, без comms.age — S5) → консоль
 │    └─ консоль: 3 деривера (operator_state/collector/power_vm) + локальные
 │       пересчёты cockpit (C1-C5) → header/чипы/F1/F2 могут расходиться
 └─ NATS events thermal/bus/pdu → ∅ потребителя (S6)
Мёртвые ветви: radar_pipeline/fusion (S1в), events (S6), app:4940 (C11).
```

## Рекомендуемый порядок починки

1. **С-быстрые честности (один срез)**: C2 (90→константа), C3 (status_by_min
   везде + `<`/`<=`), C4 (30.0→COMMS_AGE_CRIT_S), C11 (снести мёртвую ветку).
   Дёшево, убирает видимые противоречия панелей.
2. **C1+C13+C6 (срез «один владелец age/trust/SoC-ключей»)** — конвейер
   как radar_freshness.
3. **S1+S2 — отдельный дизайн-трек** (крупный): канонизировать владельца
   идентичности целей (мозговой матчер влияет на FSM), мост → транспорт
   без собственной идентичности ЛИБО консоль слушает мозговые треки;
   закрепить владельца тика (мозг №1) — intents как read-only проекция.
   Требует ADR + RAG-грунт (§17 freshness, command gating).
4. **S3+S4** — shared-владельцы свежести/env.
5. **C7** render→parse подстраниц → структурированные секции VM.
6. Гигиена: C8, C9, C10, C12, S6 (снять или подключить события).

Связь с пакетом playable: этап 9 (intent по контекстным действиям) обострит
S1 (действие по треку, которого мозг не знает под этим id) — пункт 3
желательно решить хотя бы на уровне ADR до этапа 9b (трек-сцена).
